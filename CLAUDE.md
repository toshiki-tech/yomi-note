# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

YomiNote is a Tauri 2 + React 18 + TypeScript Markdown desktop editor whose distinctive feature is built-in Japanese reading aids: furigana (kanji → hiragana) and English glosses for katakana loanwords. All frontend logic lives in `src/`; the Tauri/Rust shell lives in `src-tauri/`.

## Commands

```powershell
npm install              # also runs patch-package via postinstall
npm run tauri:dev        # full app (Rust shell + Vite). First run compiles Rust crates (~minutes).
npm run tauri:build      # production build → src-tauri/target/release/bundle/
npm run dev              # Vite-only (browser; Tauri APIs will fail). Rarely useful.
npm run build            # tsc -b && vite build (typecheck + bundle, no Rust)
npm run build:jmdict     # regenerates public/jmdict-katakana.json from JMdict-simplified
```

There is no test runner or linter configured. `npm run build` is the only typecheck.

`scripts/test-katakana-dict.mjs` is an ad-hoc verification script for the generated dictionary.

## Architecture

### Two processes, glued by Tauri events

- **Frontend** (`src/`): React + Zustand state + CodeMirror 6 editor + markdown-it preview.
- **Backend** (`src-tauri/src/lib.rs`): owns the native menu bar and right-click context menu. The menu is rebuilt on language change via the `set_menu_language` Rust command. All menu items emit a `menu-action` event with their item ID; the frontend dispatches on that ID in `src/hooks/useMenuListener.ts`. **Adding a menu item requires three coordinated changes**: (1) build it in `lib.rs` `build_app_menu` with a stable ID, (2) add labels for `en`/`ja`/`zh` in the `MenuLabels` impls, (3) handle the ID in `useMenuListener.ts`.

The frontend also intercepts the WebView2 default context menu (`oncontextmenu` in `App.tsx`) and asks Rust to popup a localized native menu via `show_context_menu`.

### State (Zustand, no Redux/Context)

Three stores, all in `src/store/`:
- `useAppStore` — open documents, active tab, view mode, sidebar/dialog flags, recent files, file tree.
- `useSettingsStore` — user preferences, persisted via `tauri-plugin-store` to `yominote-settings.json` in the OS app data dir.
- `useUserDictStore` — user-defined katakana → English entries, persisted in the same store file under key `userDict`.

Stores are accessed both via React hooks and via `useXStore.getState()` from non-React code (event handlers, Tauri callbacks).

### Japanese annotation pipeline

Two layers, both feeding the same `{base|reading}` source-level syntax:

1. **Furigana** (`src/utils/japaneseAnnotate.ts`): kuroshiro + kuromoji morphological analysis. The kuromoji dictionary (~12 MB, gzipped `.dat.gz` files in `public/kuromoji/dict/`) is loaded lazily on first use.
2. **Katakana → English**: longest-match lookup against a 3-tier dictionary, priority **USER → BUILTIN → JMdict** (see `src/utils/katakanaDict.ts`). The BUILTIN tech-term map is hardcoded; the JMdict tier is generated by `scripts/build-jmdict-katakana.mjs` into `public/jmdict-katakana.json` (~6k entries) and fetched lazily.

The user dictionary lookup is wired in via a setter (`setUserDictLookup`) called from `App.tsx` to avoid a circular import between `katakanaDict.ts` and `useUserDictStore.ts`.

#### Critical workaround: kuromoji + Vite + WebView2

Vite's dev server auto-decompresses `.gz` responses (sets `Content-Encoding: gzip`), but kuromoji's `DictionaryLoader` expects raw gzipped bytes and re-decompresses them. The result: dictionary loading fails silently in dev. `patchXHRForKuromoji()` in `japaneseAnnotate.ts` monkey-patches `XMLHttpRequest.prototype.open` to detect already-decompressed responses (no `0x1f 0x8b` magic bytes) and re-gzip them with pako before kuromoji's onload fires. **Do not remove this patch** unless you also change how the dictionary is served.

There is also a `patches/kuromoji+0.1.2.patch` applied via `patch-package` on install — needed for kuromoji to work with bundlers.

### Custom `{base|reading}` ruby syntax

The source format for furigana/glosses is `{漢字|かんじ}` and `{カメラ|camera}`. This is not standard Markdown — it's an inline rule registered on markdown-it in `src/utils/markdown.ts` (see the `zen_ruby` rule, run before `emphasis`). Reading is auto-classified: ASCII-only readings get `class="zen-katakana"` (English gloss styling), otherwise default ruby (furigana). The annotation actions in `src/utils/annotateActions.ts` write this syntax into the buffer; the markdown renderer expands it to `<ruby>` at preview time.

`stripAnnotations()` removes existing `{|}` so that re-annotating a previously annotated selection doesn't double-wrap.

### markdown-it config

`html: true` is intentional — users may write `<ruby>` directly. This is acceptable because the editor only opens local files; if you ever introduce remote/untrusted markdown sources, add a sanitizer.

### Persistence

- Window state (size/position): `tauri-plugin-window-state` (automatic).
- Settings + recent files + user dict: `tauri-plugin-store` → `yominote-settings.json`. Renaming this filename loses existing local data.
- Recent files list is capped at 20 (deduped, MRU first) in `useAppStore.pushRecentFile`.

### i18n

Three languages: `en`, `ja`, `zh`. Frontend strings: `src/i18n/translations.ts` accessed via `useT()` / `t()`. Native menu labels: separate Rust `MenuLabels` struct in `src-tauri/src/lib.rs`. **Both must be updated when adding new menu items or user-facing strings.** `system` language resolution lives in `resolveLanguage()`.

### File system access

All FS goes through `src/utils/fs.ts`, which wraps `@tauri-apps/plugin-fs` and `@tauri-apps/plugin-dialog`. The Tauri capability allowlist for FS scopes lives in `src-tauri/capabilities/default.json` — currently broad (`**`, `$HOME/**`, etc.) to support arbitrary file open/save.

Path separator is detected via `navigator.platform` (sync) since some code paths can't be async. Don't replace this with `@tauri-apps/api/path` without making call sites async.

## Conventions

- Comments and identifiers are mixed Japanese/English; the codebase tilts Japanese for explanatory comments. Match the style of the file you're editing.
- Tailwind color tokens use the `zen-*` / `zen-dark-*` namespace (e.g., `bg-zen-bg`, `text-zen-dark-text`). These are design tokens, unrelated to past project naming.
- Zustand selectors are written one-per-`useStore` call, not destructured, to avoid unnecessary re-renders.
